什么是 ZK-Email？
一句话解释：
用户在浏览器里证明"我拥有这个邮箱"，但合约只知道"验证通过"，完全不知道邮箱地址是什么。

类比：

传统方式：你向保安出示身份证，保安知道你是谁
ZK-Email：你向保安出示一张"我是真实人类"的通行证，但上面没有名字

import { proveEmailOwnership } from '@zk-email/verify'

// 用户输入验证码后，在本地生成证明
async function generateZKProof(email: string, code: string) {
  
  // 显示"正在本地验算"动画
  showAnimation('正在断开网络连接，确保数据完全停留在本地...')
  
  // 🪄 魔法发生在这里：
  // 浏览器在本地进行复杂的数学运算
  const proof = await proveEmailOwnership({
    email,
    code,
    // 私有输入：只有浏览器知道，合约看不到
    privateKey: generateLocalKey()
  })
  
  showAnimation('邮箱地址已在本地物理粉碎 🗑️')
  
  return proof
  // proof 包含：
  // - publicSignals: ["验证通过"] ← 只有这个会发给合约
  // - proof: 密码学证明 ← 合约可以验证，但反推不出邮箱
}

import { zkEmailVerifier } from "./ZKEmailVerifier.sol"

contract SanctuaryProtocol {
    using zkEmailVerifier for zkEmailVerifier.Proof;
    
    // 验证ZK证明
    function verifyAndClaim(
        bytes calldata proof,        // ZK证明
        uint256[2] calldata publicSignals  // 公共信号
    ) external {
        // 验证证明是否有效
        require(
            zkEmailVerifier.verifyProof(proof, publicSignals),
            "无效的邮箱证明"
        );
        
        // 检查是否已领取
        require(!hasClaimed[msg.sender], "已经领取过了");
        
        // 发放资金
        hasClaimed[msg.sender] = true;
        payable(msg.sender).transfer(AID_AMOUNT);
        
        emit AidClaimed(msg.sender, block.timestamp);
    }
}

## 📚 小白用户指南：IPFS 功能什么时候用？
### 什么时候会用到？
场景 说明 写日记时 每次写完 OH 卡日记点击"封印上链"，内容会自动加密并上传到 IPFS 查看历史日记 在"我的庇护所"页面查看过往记录时，会从 IPFS 下载并解密 换设备访问 只要有钱包，在任何设备上都能访问自己的日记（因为数据在 IPFS 上）

### 一句话解释 IPFS 就是你的"去中心化云盘" —— 日记不是存在某个公司的服务器上，而是存在全球分布的节点网络中，只有你能解密查看。
## ⚠️ 使用注意事项
### 1. 钱包必须连接
- 写日记前需要先连接钱包
- 加密密钥是从钱包签名派生的，没有钱包无法加密/解密
### 2. 不要丢失钱包
- 如果丢失了钱包私钥， 永远无法解密 之前的日记
- 建议备份钱包助记词
### 3. 首次需要签名
- 每次打开浏览器后，第一次加密/解密需要钱包签名
- 签名消息是固定的："Derive encryption key for OH Card Journal"
- 同一会话期间（不刷新页面）不需要重复签名
### 4. IPFS 上传需要时间
- 通常 3-10 秒完成上传
- 如果网络不好可能会失败，可以重试
### 5. 内容是加密的
- 即使有人拿到了 IPFS 上的数据，没有你的钱包也无法解密
- 比传统云盘更安全
