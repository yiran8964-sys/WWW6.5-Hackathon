关键功能链

offer 提交 / 身份验证
  • 功能目标
    • 用户提交 offer（图片/文件）→ 系统做 OCR/真实性判断 → 形成“已验证凭证”（credential），用于后续铸造 SBT / 解锁写评权限。
  • 当前实现方式（基于现有代码）
    • 后端只有“OCR 解析 offer 图片”这一段能力：把 base64Image 发给 OpenAI，让模型抽取 companyName / isValid / expireDate。
    • 没有看到“offer 上传（multer）→ 存储 → 审核/签名 → 发放 credential/signature”的完整链路实现。
    • 前端目前的 review 页面是“最小链路”调合约，不包含 offer 提交 UI；它只是提示 signature 需要脚本生成。
  • 涉及文件
    • backend_/src/services/auth.service.ts
      • 关键函数：AuthService.extractOfferInfo(base64Image)
    • backend_/src/routes/ai.routes.ts
    • backend_/src/controllers/ai.controller.ts
      • 相关接口：POST /api/v1/ai/ocr-offer（但实现调用有问题，见下）
    • backend_/src/config/openai.ts、backend_/src/config/env.ts
    • （依赖但缺失）backend_/src/types/auth.types.ts：OfferOCRResult 类型被引用但文件不存在
    • ui/mentor-review-web3/src/app/review/page.tsx
      • 文案提示 signature 需要脚本生成（没有实际 offer 流程）
    • contracts/scripts/testMintSBT.ts
      • 展示“如何编码并用后台钱包签名”的思路（更像后端签发凭证的雏形）
  • 问题点
    • 接口实现调用错误：AIController.ocrOffer 调用的是 AIService.extractOfferInfo，但 backend_/src/services/ai.service.ts 并没有这个方法；真正的 OCR 在 AuthService.extractOfferInfo。
      • 文件：backend_/src/controllers/ai.controller.ts + backend_/src/services/ai.service.ts + backend_/src/services/auth.service.ts
    • 类型缺失：OfferOCRResult 引用的 backend_/src/types/auth.types 不存在，TS 编译/运行会直接报错。
    • “身份验证”未闭环：OCR 只返回“看起来像/不像 offer”，没有把它变成链上可验证的 credentialHash、credentialId、signature 的发放流程。
  • 缺失点
    • offer 上传入口（路由/controller）与文件接收（虽有 multer 依赖，但代码未使用）
    • OCR 结果的“审核决策/风控规则/人工复核”
    • 后端签发凭证：生成 credentialId/companyId/credentialHash/expireTime/signature
    • 前端 offer 提交页面与调用后端 API 的逻辑
SBT 铸造
  • 功能目标
    • 只有通过身份验证的用户才能铸造“实习/在职凭证 SBT”，用于后续“能否查看解密内容/能否提交 review”的门控。
  • 当前实现方式
    • 前端在 review 页面直接调用合约 mintSBT(...)（wagmi useWriteContract）。
    • mintSBT 参数里包含 _signature，代码/注释明确这是 off-chain 签名（通常由后端或签发者生成），用来证明用户的 credential 有效。
    • 仓库里提供了 脚本演示如何生成签名并铸造（contracts/scripts/testMintSBT.ts），但后端并没有“签名发放 API”。
  • 涉及文件
    • ui/mentor-review-web3/src/lib/contract.ts
      • 关键常量：internSbtAddress、internSbtAbi（包含 mintSBT）
    • ui/mentor-review-web3/src/app/review/page.tsx
      • 关键调用：writeContract({ functionName: "mintSBT", args: [...] })
      • 关键辅助：bytes32FromTextOrHex()（把输入转成 bytes32：若非 32 字节 hex 则 keccak256(toBytes(v))）
    • contracts/scripts/testMintSBT.ts
      • 关键逻辑：对 ["string","address","string","bytes32","uint256"] 做 ABI encode → keccak256 → backendWallet.signMessage(bytes(hash)) → 调 mintSBT
    • contracts/ignition/deployments/fuji-20260331-v2/deployed_addresses.json
      • 部署地址与前端 contract.ts 一致（Fuji v2）
  • 当前是否真实可用还是占位/mock
    • 链上层面：大概率可用（有真实部署地址、有脚本可跑、有前端写合约调用）。
    • 产品链路层面：占位/半成品（signature 的发放目前靠脚本手动生成，缺少“身份验证 → 发放签名”的后端闭环）。
  • 问题点
    • “后端签发者”缺位：没有 API 给前端拿 _signature。
    • credentialId/companyId/credentialHash/expireTime 的规范/来源未定义（前端目前是手填）。
    • 后端的 ContractService 用的是 viem + sepolia（backend_/src/config/contract.ts），而前端/合约部署是 Avalanche Fuji(43113)，链配置不一致。
  • 缺失点
    • 后端签名服务（把 offer/OTP 验证结果转成可铸造的签名）
    • Fuji 链读取/校验的后端配置（如果后端也要读链/验 SBT）
review 正文上传到 IPFS 并拿到 CID
  • 功能目标
    • 把 review 正文（隐私内容）加密后上传 IPFS（Pinata）→ 返回 CID → 后续上链只存 CID（或 CID 的哈希/bytes32）。
  • 当前实现方式
    • 后端实现了完整的“加密 + Pinata pinJSONToIPFS”逻辑，并能返回 cid 与 ipfsUrl。
    • 但目前 没有路由/controller 对外提供“上传 review → 返回 CID”的 API。
    • 前端 review 页面里 cidInput 默认是占位字符串，并在上链前被 keccak256 成 bytes32；并没有调用后端 IPFS 上传。
  • 涉及文件
    • backend_/src/services/ipfs.service.ts
      • 关键函数：IPFSService.uploadEncryptedReview(rawContent) → pinataClient.pinJSONToIPFS(...) → 返回 uploadResult.IpfsHash
    • backend_/src/utils/encryption.util.ts
      • 关键函数：encryptContent() / decryptContent()（AES，key 来自 ENCRYPTION_KEY）
    • backend_/src/config/pinata.ts、backend_/src/config/env.ts
      • 关键变量：PINATA_API_KEY/PINATA_API_SECRET、ENCRYPTION_KEY
    • ui/mentor-review-web3/src/app/review/page.tsx
      • cidInput（占位）→ cid = bytes32FromTextOrHex(cidInput)（本质是 hash，并非真实 CID bytes32 编码）
  • 当前是否真实可用还是占位/mock
    • 后端服务代码层面：逻辑可用（只要 env 配齐且 Node 运行环境支持 fetch）。
    • 项目联调层面：当前不可用/未接通（无 API 暴露、前端也没调用）。
  • 问题点
    • getDecryptedReview 直接用 fetch(gateway.pinata.cloud/ipfs/${cid})，运行时依赖 Node 的 fetch（Node18+ 才内置）。
    • 加密 key 长度硬要求 32（env.ts），否则启动直接退出。
    • 上链使用的是 bytes32 _cid，但后端返回的是 CID 字符串；需要一个明确的“CID → bytes32”的规则（现在前端用 keccak256(ipfs://...)，属于“哈希引用”，不是原始 CID）。
  • 缺失点
    • POST /api/v1/ipfs/upload（或类似）路由/controller
    • “CID(bytes32) 规则”的统一（前端/后端/合约必须一致）
    • 与 submitReview 的打通（上传后把得到的 CID/哈希传给合约调用）
submitReview 调用合约并把 CID/哈希写到链上
  • 功能目标
    • 用户在持有有效 SBT 的前提下，提交 review：把 target（mentor/company）与评分维度、CID（或其哈希）写到链上，供后续声誉聚合/展示。
  • 当前实现方式
    • 前端直接调用 ReviewContract.submitReview(...) 写链。
    • CID 在前端被处理为 bytes32：如果不是 32-byte hex，就 keccak256(toBytes(value))。
    • 合约侧真实调用也有脚本验证（contracts/scripts/testSubmitReview.ts）。
  • 涉及文件
    • ui/mentor-review-web3/src/lib/contract.ts
      • 关键 ABI：reviewContractAbi（submitReview(uint256, bytes32, string, uint8, uint8[5], bytes32)）
      • 关键地址：reviewContractAddress
    • ui/mentor-review-web3/src/app/review/page.tsx
      • 关键调用：writeContract({ functionName: "submitReview", args: [...] })
      • 关键处理：targetId、cid 都走 bytes32FromTextOrHex()
    • contracts/scripts/testSubmitReview.ts
      • 关键逻辑：targetId = keccak256("mentor:0x1234")、cid = keccak256("ipfs://QmTest123") → submitReview(...)
    • backend_/src/services/contract.service.ts（注意：这是“读取链上数据”的服务，并不负责 submitReview）
      • 现状：ABI 是占位注释版 + 函数全是 readContract（无写入）
  • 当前是否真实可用还是占位/mock
    • 链上写入：可用（前端/脚本都能发交易，且有真实部署地址）。
    • 与 IPFS/后端的完整业务闭环：占位（CID 目前手填/占位，没有“上传 → 得到 CID → 统一转换 → 上链”的自动链路）。
  • 问题点
    • 后端 ContractService 的 ABI/函数名与前端/合约不一致：后端用 hasValidSBT/getUserSBT/getCompanyReviews 这套“字符串 companyName + ipfsCid(string)”接口，更像另一版合约；而前端合约是 isValidCredential / submitReview(bytes32 _cid) 这套。
    • 后端链配置是 sepolia（backend_/src/config/contract.ts），与前端 Fuji 不一致。
    • bytes32 _cid 实际存的是“哈希引用”，未来如果要可追溯地还原 CID，需要额外的映射或约定（例如链上存 multihash 的另一种编码/或链下数据库存储）。
  • 缺失点
    • 统一的“目标 ID / CID bytes32”规范（现在前端用 keccak256 文本，后端 IPFS 返回的是 CID 字符串）
    • submitReview 前的前置校验/组装（比如自动填维度分数、从 AI 提取结果映射到 uint8[5]）
    • 后端（如果要做声誉看板）对 Fuji 链的读取与事件索引/聚合实现（目前 ReputationService 基于另一套合约读取形状）
